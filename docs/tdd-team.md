# tdd-team

에이전트 기반 Red-Green-Refactor TDD 사이클을 자동으로 실행합니다.  
`red`, `green`, `refactor` 세 에이전트가 순차적으로 실행되며 한 번에 하나의 동작 단위를 구현합니다.

## 언제 사용하나요?

- TDD 사이클을 에이전트에게 위임하고 싶을 때
- PRD나 계획 문서가 있고, 테스트 우선으로 기능을 구현할 때
- 단계별로 구현하면서 각 사이클에서 피드백을 받고 싶을 때

> 구현 후 테스트를 작성하거나, 기존 테스트를 실행하거나, 테스트 실패를 디버깅하는 용도로는 사용하지 않습니다.

## 트리거 문구

```
"TDD로 개발해줘"
"TDD로 구현해줘"
"TDD 시작"
"TDD 팀 만들어"
"테스트 주도 개발"
"테스트 먼저 작성하고 싶어"
"테스트부터 짜줘"
"이 기능 테스트 먼저 만들고 하나씩 구현하자"
```

## 실행 흐름

### 준비 단계

1. **도메인 불변성 도출** — PRD 또는 요구사항에서 비즈니스 규칙 추출

```
도메인 불변성 (비즈니스 규칙)

  ┌─────┬─────────────────────────────────────────────────────┐
  │  #  │                       불변성                        │
  ├─────┼─────────────────────────────────────────────────────┤
  │ 1   │ 결제 완료 상태의 주문 금액은 변경할 수 없다            │
  ├─────┼─────────────────────────────────────────────────────┤
  │ 2   │ 취소된 주문은 재활성화할 수 없다                      │
  └─────┴─────────────────────────────────────────────────────┘
```

2. **태스크 분해** — 각 불변성을 `@DisplayName`이 될 도메인 규칙 문장으로 표현 후 사용자 확인
3. **프로젝트 컨텍스트 캡처 (1회)** — 오케스트레이터가 기능 영역을 한 번만 탐색해 `PROJECT_CONTEXT` 블록(패키지 구조, 테스트/픽스처 컨벤션, 관련 타입 시그니처, 도메인 앵커)을 캡처. 이후 RED/GREEN/REFACTOR 에이전트는 코드를 재탐색하지 않고 이 블록을 참조

### TDD 사이클 실행 (태스크당)

```
RED 에이전트
└── 실패하는 테스트 작성 + 실패 확인
    ↓
GREEN 에이전트
└── 테스트를 통과하는 최소 코드 작성
    ↓
REFACTOR 에이전트
└── 코드 품질 개선 (테스트 통과 유지)
    ↓
⛔ 체크포인트 — 사용자 승인 필수
    ↓
다음 태스크
```

### 사이클 요약 예시

```
── TDD Cycle 1 Complete ──
RED:      ✅ Test written: 결제_완료_주문은_취소할_수_없다
GREEN:    ✅ Implementation: OrderCancelService.cancel() 예외 처리 추가
REFACTOR: ✅ 메서드명 명확화, 중복 제거
Tests: 5 passed, 0 failed

[x] 1. 취소 불가 검증   [>] 2. 환불 금액 계산   [ ] 3. 이벤트 발행
```

## 실행 방식

- RED / GREEN / REFACTOR / Cycle Reviewer / Final Reviewer 단계 모두 서브에이전트로 위임
- 서브에이전트 도구를 사용할 수 없는 경우 `not available`로 표시 후 로컬 실행으로 전환
- 프로젝트 CLAUDE.md에 "각 단계 후 피드백 요청" 규칙이 있으면 RED → GREEN → REFACTOR → 리뷰 단계마다 사용자 승인 대기

### 테스트 실행 최적화

사이클 시간의 대부분을 차지하던 "매 단계 전체 테스트 스위트 실행"을 제거했습니다.

- **사이클 중에는 작업 중인 테스트 클래스만 실행** (`TEST_SCOPED_CMD`, 예: `./gradlew test --tests "FQCN" --offline`)
- **전체 스위트는 Final Review에서 한 번만** 실행해 교차 클래스 회귀를 확인
- Setup에서 **빌드 데몬 워밍업 1회**, 이후 데몬 상시 유지 · `--offline` · `clean` 금지로 콜드스타트 제거
- **프로젝트 컨텍스트 1회 캡처** — 매 에이전트가 코드를 재탐색하던 오버헤드를 제거해 RED/GREEN 단계 자체의 지연을 줄임

## 주의사항

- 오케스트레이터는 파일을 직접 수정하지 않고 반드시 서브에이전트를 통해 위임합니다
- 매 사이클 후 사용자 승인 없이는 다음 태스크로 진행하지 않습니다
- plan-creator 문서가 없으면 요구사항 공유를 먼저 요청합니다
