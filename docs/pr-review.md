# pr-review

Bitbucket PR을 PN 룰 기반으로 4차원 코드 리뷰하고 리포트를 저장합니다.  
PR URL만 공유하면 로컬 리포지토리를 자동으로 찾아 diff를 가져와 리뷰합니다.

## 언제 사용하나요?

- Bitbucket PR을 체계적으로 리뷰하고 싶을 때
- 버그·아키텍처·컨벤션·테스트 커버리지를 한 번에 점검하고 싶을 때
- 리뷰 코멘트에 일관된 우선순위 태그(PN 룰)를 붙이고 싶을 때

## 트리거 문구

```
"PR 리뷰해줘"
"코드리뷰해줘"
"이 PR 봐줘"
"{Bitbucket PR URL}" (URL만 붙여도 리뷰 의도로 인식)
```

## PN 룰 (우선순위 기준)

| 태그 | 의미 | 대응 |
|------|------|------|
| **[P1]** | 꼭 반영해주세요 | 머지 전 필수 수정 |
| **[P2]** | 적극적으로 고려해주세요 | 수용하거나 토론 |
| **[P3]** | 웬만하면 반영해 주세요 | 수용하거나 JIRA 티켓으로 계획 |
| **[P4]** | 반영해도 좋고 넘어가도 좋습니다 | 무시해도 무방 |
| **[P5]** | 사소한 의견 | 무시해도 됨 |

## 실행 흐름

1. **URL 파싱** — workspace / repo_slug / PR_ID 추출
2. **로컬 리포지토리 찾기** — `~/.claude/repo-dirs.md` 조회 또는 경로 탐색
3. **Diff 조회** — Bitbucket API(토큰 있을 때) 또는 git 로컬로 source/destination 확인, **`...`(세 점)** diff 사용
4. **도메인 컨텍스트 로드** — `docs/domain/{domain}.md` 존재 시 읽어서 도메인 규칙 파악
5. **테스트 커버리지 확인** — 행동 변경 코드에 대응하는 테스트 존재 여부 확인
6. **코드 리뷰** — 버그/아키텍처/컨벤션/테스트 4차원 평가, PN 태그 부착
7. **리포트 저장** — `{LOCAL_PATH}/docs/reviews/pr-{PR_ID}-review.md`

## 리뷰 5원칙

1. **의도를 먼저 파악** — 단정 전에 "왜 이렇게 짰나요?" 먼저 묻기
2. **이유와 근거 설명** — "바꾸세요"가 아닌 왜 개선이 필요한지 맥락 포함
3. **키워드·방향 제시** — 완성 코드 제시 금지, 스스로 찾게 유도
4. **스타일 강요 안 함** — 합의된 컨벤션 기준이 아닌 순수 취향 차이는 P5 이하
5. **지적할 것이 없으면 칭찬** — 코드 라인 인용해서 구체적으로

## 출력 예시

```
| 우선순위 | 파일           | 요약               | 카테고리 |
|----------|----------------|--------------------|----------|
| P1       | OrderService:42 | 트랜잭션 경계 불명확 | 아키텍처 |
| P2       | PaymentDto:18   | 테스트 커버리지 없음 | 테스트   |
| 의도확인 | CouponService:7 | 왜 null 허용인가요? | —        |
```
