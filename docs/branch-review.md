# branch-review

현재 브랜치의 모든 변경사항을 4개 차원으로 분석하고 점수화합니다.  
PR 생성 전 코드 품질을 확인하거나 코드 리뷰를 준비할 때 사용합니다.

## 언제 사용하나요?

- PR을 올리기 전 코드 품질을 사전 점검할 때
- 팀 컨벤션 준수 여부를 자동으로 확인하고 싶을 때
- 리뷰어 입장에서 개선 우선순위를 파악하고 싶을 때

## 트리거 문구

```
"브랜치 리뷰해줘"
"코드 리뷰"
"PR 전에 리뷰해줘"
```

## 평가 4개 차원

| 차원 | 가중치 | 주요 평가 항목 |
|------|--------|---------------|
| A. 코드 컨벤션 | 25% | 도메인 접두사 네이밍, DTO 패턴, `@RequiredArgsConstructor`, 명시적 타입 선언, 4개 이상 파라미터는 객체로 |
| B. 테스트 품질 | 30% | AAA 패턴 주석, `@Nested` 구조, `@DisplayName` 한국어, SUT 명명 |
| C. 도메인 로직 | 30% | 비즈니스 규칙 정확성, `@Transactional` 설계, 예외 타입, enum 메서드 캡슐화 |
| D. 설계 품질 | 15% | 단일 책임, 공유 모듈 재사용, 유지보수성, 코드 스멜 감지 |

### 등급 기준

```
A: 90+   B: 80+   C: 70+   D: 60+   E: 50+   F: 50 미만
종합 = (A×0.25) + (B×0.30) + (C×0.30) + (D×0.15)
```

## 코드 스멜 감지 항목

- **중복 코드** → 누락된 추상화
- **대형 클래스** → 책임 분리 필요
- **데이터 뭉치** → 클래스 추출
- **Feature Envy** → 메서드가 다른 클래스의 데이터를 더 많이 사용
- **메시지 체인** `a.b().c().d()` → Law of Demeter 위반
- **산탄총 수술** → 하나의 변경이 여러 파일에 흩어짐
- **투기적 일반화** → YAGNI 위반
- **타입별 switch/if-else** → 다형성 또는 전략 패턴으로 대체

## 유지보수성 체크

- 메서드 ~30줄 초과 → 혼재된 책임 가능성
- 중첩 3단계 초과 → 조기 반환 또는 메서드 추출 고려
- 이름만으로 의도 전달 안 됨 → 주석 없이도 읽히는 네이밍
