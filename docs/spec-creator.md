# spec-creator

대규모 기능의 기술 명세(Spec) 문서를 작성합니다.  
전체 범위와 도메인 컨텍스트를 정의하고, plan-creator가 처리할 수 있는 독립적인 하위 태스크로 분해합니다.

## 언제 사용하나요?

- 여러 개발 세션에 걸쳐 구현해야 할 큰 기능의 기술 명세가 필요할 때
- 도메인 컨텍스트, 비즈니스 불변성, 하위 태스크 분해를 한 문서로 정리하고 싶을 때
- devlife-brainstorming으로 방향을 잡은 뒤 기술 세부사항을 작성할 때

## 트리거 문구

```
"스펙 작성해줘"
"spec 작성해줘"
"기술 명세 작성해줘"
"큰 기능 계획해줘"
"에픽 스펙 만들어줘"
```

## 실행 흐름

1. **문서 입력 읽기** (선택) — brainstorming 문서가 있으면 먼저 읽고 기존 결정사항 파악
2. **코드베이스 탐색** — 영향 받는 도메인, 서비스, 컨트롤러 파악
3. **정보 수집** — 배경/목표, 도메인 컨텍스트, 비즈니스 불변성, 범위, 하위 태스크 분해 방식에 대해 **4개씩** 질문 (라운드 단위)
4. **Spec 문서 작성** — `spec-{feature}.md` 생성 (프로젝트 루트)
5. **자동 검토** — 독립 서브에이전트가 내용 검토 후 이슈 수정
6. **피드백 요청** — 하위 태스크 분해 방식, 범위, 우선순위 검토

## 생성 문서 구조

```
spec-{feature}.md
├── 배경 및 목표
├── 도메인 컨텍스트    ← 산문체로 작성 (bullet 아님)
├── 비즈니스 불변성    ← "X 상태에서는 Y가 절대 불가능하다" 형식
├── 범위 (in-scope / out-of-scope)
├── 하위 태스크 목록 (T1, T2, ...)
├── 태스크 크기 및 의존관계
└── 성공 기준
```

## 태스크 크기 기준

| 크기 | 기준 |
|------|------|
| S (Small) | 단일 파일 변경, 3시간 이내 |
| M (Medium) | 2-5개 파일, 반나절~하루 |
| L (Large) | 여러 모듈, 하루 이상 → 더 쪼개기 |

## 핵심 원칙

- **추측 금지** — 도메인 컨텍스트나 불변성을 모르면 질문. 추론으로 채우지 않음
- **불변성은 완전한 문장으로** — "결제 완료 상태의 주문 금액은 어떤 경우에도 변경할 수 없다. 허용 시 정산 불일치 및 회계 감사 실패로 이어진다."
- **M 이하로 분해** — L 크기 태스크는 반드시 더 작게 쪼갬

## 워크플로우 위치

```
devlife-brainstorming (선택)
        ↓
  spec-creator  ← 현재 위치
        ↓

```
