## 우선순위 레벨

| 태그 | 의미 | 대응 방식 |
|------|------|-----------|
| **P1** | 꼭 반영해주세요 | 중대한 오류를 발생할 수 있는 가능성 — 머지 전 필수 수정 |
| **P2** | 적극적으로 고려해주세요 | 수용하거나, 수용할 수 없으면 적합한 의견을 들어 토론 권장 |
| **P3** | 웬만하면 반영해 주세요 | 수용하거나, 못하면 반영 불가 이유 설명 또는 JIRA 티켓으로 계획 명시 권장 |
| **P4** | 반영해도 좋고 넘어가도 좋습니다 | 무시해도 무방. 반영할지 고민해보는 정도면 충분 |
| **P5** | 그냥 사소한 의견입니다 | 무시해도 무방 |

## 코멘트 작성 원칙

### 1. 이유와 근거를 설명한다

단순히 "이 코드가 잘못됐다"가 아닌, **왜** 문제인지 맥락과 근거를 함께 서술한다.

예시:
> "해당 코드에서 이중 조건문의 경우, 시스템에 복잡성을 추가하고 있습니다.
> 또한, 빈 입력 배열이나 항목이 어떤 조건도 충족하지 않을 때와 같은 특이 사례를
> 커버하기 어려워보여요. XXX 방식으로 처리하는 게 성능 향상과 특이 케이스 처리에서
> 유리합니다."

### 2. 정답보다 방향성을 제시한다

"이렇게 바꾸세요"가 아닌, 고려해볼 수 있는 방향을 제안한다.

예시:
> "Java에는 Stream이라는 것이 존재하는데요,
> 해당 코드에서는 map을 사용해보는 건 어떨까요?
> map을 사용하면 코드량을 줄이고 가독성을 높일 수 있을 것 같습니다."

## P1이 적용되는 전형적인 케이스

- 비즈니스 로직 오류 (잘못된 조건, 누락된 분기)
- 트랜잭션 롤백 위험 (`@Transactional` + catch 충돌)
- Null pointer 가능성이 있는 미처리 경로
- 데이터 정합성 문제 (잘못된 저장, 덮어쓰기)
- 보안 취약점

## P2가 적용되는 전형적인 케이스

- 아키텍처 레이어 위반 (Controller에서 DB 직접 접근 등)
- 성능 문제 (N+1 쿼리, 불필요한 전체 조회)
- 테스트 커버리지 누락 (핵심 비즈니스 케이스)

## P3가 적용되는 전형적인 케이스

- 컨벤션 위반 (네이밍, 패키지 구조)
- 코드 스멜 (중복, 긴 메서드, 복잡한 조건)
- DI 방식 불일치 (@Autowired 필드 주입)

## P4/P5가 적용되는 전형적인 케이스

- 스타일 제안 (람다 → 메서드 레퍼런스)
- 변수명 개선 제안
- 주석 추가 제안
