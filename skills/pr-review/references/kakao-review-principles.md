## 리뷰의 5가지 규칙

### 1. 왜 개선이 필요한지 이유를 충분히 설명한다

피드백이 주관적이거나 추상적이면 리뷰이는 혼란을 느낀다.
리뷰이가 개선의 필요성을 스스로 느낄 수 있도록 구체적인 이유를 작성한다.

| 구분 | 예시 |
|------|------|
| ❌ 나쁜 리뷰 | "data 변수 말고 다른 변수명으로 하세요." |
| ✅ 좋은 리뷰 | "data라는 이름은 현재 자료구조의 의도를 알기가 어렵네요. 학점 정보를 담고 있다면 이와 관련된 변수명을 짓는다면 의도를 쉽게 파악할 수 있을 것 같아요." |

### 2. 답을 알려주기보다 스스로 고민하고 개선 방법을 선택할 수 있게 한다

완성된 코드를 제시하면 리뷰이는 복붙만 하고 동작 방식을 이해하지 못한다.
비슷한 코드가 여러 군데 있어도 이곳만 수정하게 된다.
키워드와 방향을 주어 스스로 찾아보고 학습할 기회를 만든다.

| 구분 | 예시 |
|------|------|
| ❌ 나쁜 리뷰 | "`arr.filter(item => item % 2 === 0);`으로 사용하세요." |
| ✅ 좋은 리뷰 | "자바스크립트 배열에는 다양한 내장 메서드들이 존재합니다. `filter` 메서드를 찾아서 활용해보세요. 코드량을 줄일 수 있을 것 같습니다." |

### 3. 코드를 클린하게 유지하고 일관되게 구현하도록 안내한다

개별 스니펫만 보지 않고 **전체 코드를 보며 일관성**을 확인한다.
함수 선언 방식, 반복문 방식 등이 혼재하면 전체 맥락에서 지적해야 한다.

| 구분 | 예시 |
|------|------|
| ❌ 나쁜 리뷰 | "함수 표현식을 사용하셨군요. reduce 사용도 잘 활용했네요." (전체 맥락 미확인) |
| ✅ 좋은 리뷰 | "함수를 선언하는 두 가지 방식을 사용했는데, 특별한 이유가 아니라면 일관된 방식을 정해보세요. calculateEarningRate에서도 reduce와 유사한 고차 함수를 찾아보면 일관성을 유지할 수 있을 것 같습니다." |

### 4. 리뷰 과정이 숙제검사가 아닌 학습과정으로 느껴지게 한다

먼저 **리뷰이의 의도를 파악**하고, 결론을 단정하기 전에 이유를 물어본다.
리뷰이가 확장성이나 특별한 이유로 그렇게 짰을 수 있기 때문이다.

| 구분 | 예시 |
|------|------|
| ❌ 나쁜 리뷰 | "클래스 상속은 필요 없습니다." |
| ✅ 좋은 리뷰 | "자식 클래스에서 부모 클래스를 호출만 하기 때문에 중복 코드처럼 보이기도 하는데, 이렇게 작성하신 이유가 있을까요?" |

### 5. 리뷰를 위한 리뷰를 하지 않는다. 피드백할 게 없으면 칭찬한다

흔적을 남기기 위해 억지로 리뷰하면 리뷰이가 잘 짜인 코드도 잘못됐다고 오해할 수 있다.
진짜로 잘 짜인 코드에는 **구체적인 칭찬**을 한다.

칭찬 예시:
- "코드량이 적당해서 읽기 편하네요."
- "예외 처리가 꽤 꼼꼼해서 좋네요."
- "함수, 변수명이 직관적이어서 좋네요."
- "성능에 아주 유리한 코드라고 생각되네요."

## 추가 팁

### 커밋 단위가 아닌 전체 코드 맥락을 본다

작은 단락에서는 깔끔해 보여도 전체 코드 흐름에서 어색하거나,
작은 단락에서 드러나지 않은 이슈가 전체에서 확인될 수 있다.
PR 전체를 읽고 diff를 본 뒤 리뷰한다.

### 개발자의 성향을 리뷰하지 않는다

추가 연산 없이 단순 값 반환이라면, `if/else`와 삼항 연산자는 가독성이 주관적이다.
팀 컨벤션으로 합의된 것이 아니라면 개인 취향 강요는 금물.
충분한 논의 없이 스타일을 강제하는 피드백은 지양한다.
